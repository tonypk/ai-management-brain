# P4: Dynamic AI Role System Design

## Goal

Replace the hardcoded AI role registry with a dynamic system where mentors freely invent roles and assign action primitives based on each company's profile. Different mentors may recommend completely different role compositions for the same company.

## Architecture

The system has three layers:
1. **Action Primitives** â€” predefined executable operations (chase, summarize, alert, etc.)
2. **Dynamic Role Definitions** â€” generated by OrgEngine/LLM as part of ManagementPlan
3. **Role Agent Runtime** â€” executes primitives on schedule/events with role-specific personality

## Action Primitives

Predefined operations that map to existing services:

| Primitive | Description | Backing Service | Mode |
|-----------|-------------|-----------------|------|
| `chase_missing` | Chase employees who haven't submitted reports | `report.Chaser` | auto_execute |
| `daily_summary` | Generate daily team status summary | `report.Summarizer` | auto_execute |
| `weekly_summary` | Generate weekly operations review | `report.ActionExecutor` | auto_execute |
| `check_alerts` | Check alert conditions and create suggestions | `report.AlertChecker` | auto_execute |
| `create_suggestion` | Generate strategic suggestion for boss approval | AI Suggestions table | suggest_only |
| `send_branded_msg` | Send role-branded message to boss | `BossSender` | auto_execute |

Each primitive is registered in an `ActionRegistry` with its execution function.

## Dynamic Role Definition (LLM Output)

OrgEngine prompt includes the available primitives list. The LLM generates `support_roles` in ManagementPlan:

```json
{
  "title": "Chief Operating Officer",
  "title_en": "Chief Operating Officer",
  "role_id": "ai-ops-director",
  "type": "ai",
  "scope": "Daily operations monitoring, team follow-up, anomaly detection",
  "personality": "Data-driven, pragmatic, detail-oriented",
  "capabilities": [
    {"action": "daily_summary", "schedule": "0 8 * * *"},
    {"action": "chase_missing", "schedule": "30 17 * * *"},
    {"action": "weekly_summary", "schedule": "0 18 * * 5"},
    {"action": "check_alerts", "trigger": "alert.fired"},
    {"action": "create_suggestion", "trigger": "alert.fired"}
  ]
}
```

Key fields:
- `role_id`: LLM-generated unique identifier (prefixed with `ai-`)
- `title` / `title_en`: Display name (LLM chooses freely)
- `scope`: Role's responsibility description (used in system prompt)
- `personality`: Role's communication style (used in system prompt)
- `capabilities[].action`: Must be a valid primitive from ActionRegistry
- `capabilities[].schedule`: Cron expression for periodic execution
- `capabilities[].trigger`: Event name for event-driven execution

## Changes

### 1. `internal/roles/registry.go` â€” Rewrite

**Before**: Hardcoded `Registry` map with `ai-coo` RoleDefinition containing capabilities.

**After**: `ActionRegistry` â€” maps primitive names to executor functions.

```go
type ActionFunc func(ctx context.Context, agent *RoleAgent) error

var ActionRegistry = map[string]ActionFunc{
    "chase_missing":    executeChaseMissing,
    "daily_summary":    executeDailySummary,
    "weekly_summary":   executeWeeklySummary,
    "check_alerts":     executeCheckAlerts,
    "create_suggestion": executeCreateSuggestion,
    "send_branded_msg": executeSendBrandedMsg,
}
```

Remove `RoleDefinition`, `Capability`, `CapabilityMode` types. Remove `Registry` map.

### 2. `internal/roles/agent.go` â€” Adapt to Dynamic Config

**Before**: Looks up `Registry[roleID]` for capabilities.

**After**: Reads capabilities from `DynamicRoleConfig` (loaded from DB `config` JSONB).

```go
type DynamicRoleConfig struct {
    Title       string               `json:"title"`
    TitleEn     string               `json:"title_en"`
    RoleID      string               `json:"role_id"`
    Scope       string               `json:"scope"`
    Personality string               `json:"personality"`
    Capabilities []DynamicCapability  `json:"capabilities"`
}

type DynamicCapability struct {
    Action   string `json:"action"`
    Schedule string `json:"schedule,omitempty"`
    Trigger  string `json:"trigger,omitempty"`
}
```

`SystemPrompt()` combines: mentor philosophy + role scope + role personality.

`RunCapability(ctx, actionName)` looks up `ActionRegistry[actionName]` and executes.

### 3. `internal/roles/capabilities.go` â€” Keep, Rename

Keep existing capability implementations but rename to match primitive names:
- `executeDailySummary` (was daily_status_check logic)
- `executeChaseMissing` (was chase_missing_reports logic)
- `executeWeeklySummary` (was weekly_summary logic)
- `executeCheckAlerts` (was detect_anomalies logic)
- Add `executeCreateSuggestion` â€” generic LLM suggestion generation
- Add `executeSendBrandedMsg` â€” simple branded message send

### 4. `internal/roles/manager.go` â€” Adapt Activation

`ActivateForTenant()`:
- Iterate `plan.OrgDesign.SupportRoles` where `Type == "ai"`
- Validate each capability's `action` exists in `ActionRegistry`
- Skip unknown actions with warning log (don't fail)
- Upsert `ai_role_instances` with full config as JSONB
- Create `RoleAgent` with `DynamicRoleConfig`
- Register cron jobs from `capabilities[].schedule`
- Subscribe to events from `capabilities[].trigger`

`LoadExistingForTenant()`:
- Load from DB, unmarshal `config` to `DynamicRoleConfig`
- Re-register cron + events

### 5. `internal/brain/org_engine.go` â€” Enhanced Prompt

Add to `buildOrgSystemPrompt()`:
- List of available action primitives with descriptions
- Instruction: "Based on the company profile and your management philosophy, define AI support roles. Each role needs: role_id (prefixed with ai-), title, scope, personality, and capabilities from the available primitives."
- Constraint: capabilities must use valid action names and cron expressions

### 6. `internal/brain/org_wizard.go` â€” No Change

Wizard collects company profile. Role generation happens in OrgEngine.

### 7. `internal/api/org_handlers.go` â€” Minor

`handleActivatePlan` already calls `roleManager.ActivateForTenant()`. No change needed since Manager handles the new dynamic format internally.

### 8. Frontend â€” No Change

`AIRolesView.vue` already displays roles dynamically from API response. Role title and capabilities come from DB, so new roles appear automatically.

## Database

No schema changes. `ai_role_instances.config` JSONB already stores the role configuration. The new `DynamicRoleConfig` struct just has richer content.

## Non-Goals

- Custom user-defined primitives (only predefined ones)
- Role-to-role communication (each role operates independently)
- Frontend role editor (roles are generated by mentor, not manually created)
